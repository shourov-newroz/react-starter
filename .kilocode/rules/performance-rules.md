# Performance Rules

These rules define performance optimization standards for the application.

---

## Code Splitting

- ALL page components MUST use `React.lazy()` for code splitting
- Routes are configured with lazy imports:

```typescript
element: React.lazy(() => import('@/features/auth/pages/LoginPage'))
```

- Every lazy-loaded component must be wrapped in `<Suspense fallback={route.fallback || <GenericPageSkeleton />}>>`

---

## Memoization

### `useCallback`

- Use for function references passed as props or dependencies
- Required when functions are used in `useEffect` / `useMemo` dependency arrays
- Pattern from codebase:

```typescript
const login = useCallback(
  async (email: string, password: string): Promise<User> => {
    // ...
  },
  [setUser, setLoading, setError]
);
```

### `useMemo`

- Use for expensive computations that don't need to run every render
- Only when the computation cost justifies the memoization overhead
- Do NOT use `useMemo` for simple object/array creation unless it causes measurable re-renders

### When NOT to memoize

- Simple variable assignments
- Cheap calculations
- Values that change every render anyway
- **Do not prematurely optimize** â€” only memoize when there's a measurable performance impact

---

## Zustand Selector Optimization

- ALWAYS use individual selectors when consuming store state:

```typescript
// âś… Correct â€” only re-renders when `user` changes
const user = useAuthStore(authSelectors.user);
const isLoading = useAuthStore(authSelectors.isLoading);

// âťŚ Wrong â€” re-renders on ANY state change
const { user, isLoading } = useAuthStore();
```

- NEVER destructure the entire store â€” always use selectors

## Bundle Splitting

Production build (`vite.config.ts`) uses manual chunks:

| Chunk | Libraries |
|---|---|
| `react-vendor` | `react`, `react-dom`, `react-router-dom`, `react-error-boundary` |
| `ui-vendor` | `zustand`, `swr`, `axios`, `clsx`, `tailwind-merge`, `class-variance-authority` |
| `radix-ui` | `@radix-ui/react-label`, `@radix-ui/react-slot` |

### Rules

- When adding significant new dependencies, consider creating new vendor chunks
- Keep chunk sizes under `1000KB` (warning limit in Vite config)
- Monitor bundle size via `dist/bundle-report.html` (generated by rollup-plugin-visualizer)

---

## SWR Optimization

SWR is pre-configured with performance optimizations:

| Setting | Value | Impact |
|---|---|---|
| `revalidateOnFocus` | `false` | No refetch when tab regains focus |
| `dedupingInterval` | `2000` | Dedup identical requests within 2s |
| `focusThrottleInterval` | `5000` | Throttle focus revalidation to 5s |

- Do NOT add manual request deduplication â€” SWR handles it
- Do NOT override global SWR config unless feature-specific behavior is required

## CSS Performance

- **TailwindCSS v4** handles tree-shaking automatically
- Use Tailwind utility classes exclusively â€” avoid custom CSS unless absolutely necessary
- Use the `cn()` utility from `@/lib/utils.ts` for conditional class names:

```typescript
import { cn } from '@/lib/utils';
<div className={cn('base-class', isActive && 'active-class')} />
```

---

## Re-Render Prevention

- Avoid inline object/array literals in JSX props (creates new references each render):

```typescript
// âťŚ Creates new object every render
<Component style={{ color: 'red' }} />

// âś… Define outside component or use useMemo
const style = { color: 'red' };
<Component style={style} />
```

- Avoid inline arrow functions for event handlers when performance-critical:

```typescript
// âťŚ Creates new function every render
<button onClick={() => handleClick(id)} />

// âś… Use useCallback or bind in advance
const handleItemClick = useCallback(() => handleClick(id), [id]);
<button onClick={handleItemClick} />
```

---

## Image Optimization

- Use optimized formats: **WebP** or **AVIF** when possible
- Lazy-load images below the fold
- Include `width` and `height` attributes to prevent layout shifts
- Use `loading="lazy"` for non-critical images

---

## Bundle Analysis

- Production builds automatically generate `dist/bundle-report.html`
- Review the bundle report after adding new dependencies
- Flag any single chunk exceeding 500KB for review

---

## Skeleton Loading Performance

- Skeleton renders synchronously â€” no delay, no spinner
- Skeleton must not cause layout shift (CLS = 0)
- Skeleton must not introduce scrolling
- Use route-specific skeleton fallback instead of generic fallback for better UX
